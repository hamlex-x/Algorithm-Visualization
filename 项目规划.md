# 命名规范
## 函数
1. 动词明确 — create/build/generate/render/execute/undo/executeAll/push/bind/mount/read/calc/init，一个动词只做一类事
2. 对象紧跟 — 动词后马上说清楚操作什么：Graph、BfsStep、Queue、CircleLayout
3. 算法名在对象后 — executeBfsStep 不是 BfsStepExecute
4. SVG/DOM 细节不出现在函数名里 — renderQueue 不给 createQueueInSVG，渲染手段是实现细节

### 动词含义

| 动词 | 含义 | 所在层 |
|---|---|---|
| `create` | 创建一个空的/默认的数据结构（"造壳"），如 `createGraph(8)` 生成 8 个节点全 0 矩阵的空图 | builder |
| `build` | 填入具体数据使对象可用（"填肉"），如 `buildGraph(matrix, graph)` 填入邻接矩阵 + 计算布局 | builder |
| `generate` | 遍历数据结构，一次性产出所有算法步骤，不操作 DOM，如 `generateBfsSteps(graph, queue)` | algo |
| `render` | 把数据初次绘制到 SVG/DOM 上，如 `renderGraph(graph)`、`renderQueue()` | renderer |
| `execute` | 执行一步 step，推进可视化（节点变色、队列变更、说明更新），如 `executeBfsStep` | renderer |
| `undo` | 反向操作，回退一步，如 `undoBfsStep` 恢复上一步之前的状态 | renderer |
| `executeAll` | 自动连续执行所有剩余步骤（内部用 timer），如 `executeAllBfsSteps` | renderer |
| `bind` | 给 DOM 元素绑定事件监听器，如 `bindBfsButtons()` | page |
| `mount` | 组装入口，调用 builder + algo + renderer 完成完整初始化，每个 page 模块必须导出此函数 | page |
| `calc` | 纯计算，无副作用，如 `calcCircleLayout(graph, radius, cx, cy)` 根据公式计算节点坐标 | builder |
| `push` | 快速写入一个 step 到 stepQueue，减少重复代码，如 `pushBfsStep(queue, "visit", 0)` | algo |
| `read` | 从 DOM 中读取用户输入，返回纯数据，不操作 DOM，如 `readGraphMatrix()` | page |
| `init` | 初始化一个 SVG 元素的属性值（相当于构造函数），如 `initRect(rect, x, y, w, h, color, fill, sw)` | shared |

# 页面导航流

先创建再使用——创建一次，可反复用不同算法观察：
过程是list → create → list → algo

- **list 页**：展示数据结构分类树 + 算法列表，点击数据结构进入创建页，点击算法进入对应算法页
- **create 页**：构建数据结构，写入 `shared/state.ts`，完成后跳回 list，
- **algo 页**：从共享状态读取数据，`build` + `render` + `generate`；若状态为空则提示"请先创建数据结构"并提供跳转链接

## 共享状态

`shared/state.ts` 作为跨页面数据中转，ES 模块在所有路由间**单例缓存**：

- create 页写入 → algo 页读取
- 一次创建后可在多个算法页之间切换复用，无需重建
- 刷新页面后重置为初始值
# 解耦原则

四个底层原理，从根上决定代码怎么分层、函数怎么写。

---

## 原理一：显式依赖 — 函数签名必须说清它需要什么

**核心规则**：函数依赖的一切外部数据，必须通过参数传入。不允许通过闭包捕获外部变量、不允许内部查询全局对象。

**反例**：

```ts
// 坏：依赖隐藏在函数体内部
function renderCategoryTreeRect() {
    const svg = document.querySelector("svg");  // 隐式依赖全局唯一的 <svg>
    // ...
}
```

```ts
// 坏：闭包捕获模块级变量
let stepIndex = 0;
function executeBfsStep() {
    stepIndex++;  // stepIndex 从哪来？看签名不知道
}
```

**正例**：

```ts
// 好：所有依赖都在参数列表中
function renderCategoryTreeRect(svg: SVGSVGElement) {
    // svg 从参数来，签名已经说清楚了
}
```

```ts
// 好：可变状态封装在对象中显式传入
function executeBfsStep(svg: SVGSVGElement, stepQueue: BfsStep[], stepIndex: number): number {
    // 输入全部从参数来，返回值是新的 stepIndex
}
```

**为什么**：隐式依赖让函数签名说谎 — 看签名以为不需要 X，实际函数内部用了 X。调用方不知道要准备什么环境。测试时也不知道要 mock 什么。

**判断标准**：只看函数签名，能不能完整写出调用代码？如果还需要看函数体才知道要准备哪些东西，就违反了显式依赖。

---

## 原理二：层隔离 — 每层只有一个职责，跨层的事不干

**四层职责边界**：

```
pages/       → 组装、绑定事件、管理模块状态、写入全局状态
renderers/   → 创建 DOM 元素、执行一步 step、撤销一步 step
algos/       → 纯计算，输入数据结构，输出 step[]
builders/    → 纯计算，输入用户数据，输出数据结构对象
shared/      → 类型定义、常量、通用工具函数、通用数据结构类
```

**核心规则**：

| 层 | 能做 | 不能做 |
|---|---|---|
| page | 读写全局状态、`document.getElementById`、绑事件 | 不能直接操作 SVG 子元素、不能写算法逻辑 |
| renderer | 创建 DOM/SVG 元素、操作传入的元素 | 不能查全局 DOM、不能绑页面事件、不能读写全局状态 |
| algo | 遍历数据生成 step | 不能操作 DOM、不能读写全局状态 |
| builder | 计算布局、构造数据对象 | 不能操作 DOM、不能读写全局状态 |

**反例**：renderer 中直接 `document.getElementById` 查元素 — 跨越了层边界，renderer 不应该知道页面中有什么元素。

**正例**：page 通过 `document.getElementById` 找到元素，作为参数传给 renderer 函数。

**为什么**：层的本质是"变更率相同的东西放一起"。算法逻辑变更时不该碰 DOM 代码，页面布局变更时不该碰算法代码。层隔离就是让每次修改的影响面最小。

---

## 原理三：渲染与行为分离 — render 和 bind 是两个阶段

**核心规则**：创建元素（render）和绑定事件（bind）必须写在两个不同的函数里。

**反例**：

```ts
// 坏：渲染函数里夹带事件绑定
function renderEnsureButton(container: HTMLElement) {
    const btn = document.createElement("button");
    btn.textContent = "确定";
    btn.addEventListener("click", () => {  // 行为逻辑混在渲染里
        graph.matrixGraph = buildGraphMatrix(readGraphMatrix());
        location.hash = "#/list";
    });
    container.appendChild(btn);
}
```

**正例**：

```ts
// 好：渲染只管创建和添加到 DOM，返回元素引用
function renderEnsureButton(container: HTMLElement): HTMLButtonElement {
    const btn = document.createElement("button");
    btn.textContent = "确定";
    container.appendChild(btn);
    return btn;
}

// 好：绑定只管查找和附加事件
function bindEnsureButton(btn: HTMLButtonElement) {
    btn.addEventListener("click", () => {
        graph.matrixGraph = buildGraphMatrix(readGraphMatrix());
        location.hash = "#/list";
    });
}

// mount 中分两个阶段调用
export function mount(container: HTMLElement) {
    const btn = renderEnsureButton(container);
    bindEnsureButton(btn);
}
```

**为什么**：
- 渲染可以独立测试 — 传入 container，检查子元素
- 行为可以独立修改 — 换事件类型、加防抖，不用动渲染代码
- 同一套渲染可以搭配不同行为 — 比如同一张表在不同场景绑不同校验规则

**判断标准**：一个函数里有没有 `addEventListener`？如果有，它就不是纯渲染函数。唯一的例外是 `executeBfsStep` / `undoBfsStep` 这类 step 执行函数，它们操作的是数据驱动的动画状态，不是页面交互行为。

## 原理四：局部 DOM 作用域 — 有引用不查全局

**核心规则**：如果已经持有某个元素的引用，就通过它做子树查询（`querySelector` / `getElementById` 的区别不在 API，在作用域）。不要从 `document` 重新查。

**反例**：

```ts
// 坏：明明有 table 引用，却绕回 document 全局查询
function bindMatrixTableValidation(graphMatrixTable: HTMLTableElement, cnt: number) {
    for (let i = 0; i < cnt; i++) {
        for (let j = 0; j < cnt; j++) {
            const input = document.getElementById(`graph_input_${i}_${j}`);  // 全局查询
        }
    }
}
```

**正例**：

```ts
// 好：从已有元素向下查，不碰 document
function bindMatrixTableValidation(graphMatrixTable: HTMLTableElement) {
    const trs = graphMatrixTable.querySelectorAll("tbody tr");
    for (let i = 0; i < trs.length; i++) {
        const tds = trs[i]!.querySelectorAll("td");
        for (let j = 0; j < tds.length; j++) {
            const input = tds[j]!.querySelector("input")!;
        }
    }
}
```

**为什么**：
- 函数的输入（`graphMatrixTable`）决定了它的作用域，不依赖页面中不能有第二个同名 id 的元素
- 如果同一个页面上有两个表格（比如对比模式），全局查询会抓错对象
- `cnt` 参数也不需要了 — DOM 结构自己知道有多少行多少列

**唯一可以查全局 DOM 的层**：page 层。因为 page 的职责就是"把页面组装起来"，它需要知道页面上有什么元素。即使如此，查完之后应该立即把引用传给 renderer/algo/builder，而不是让它们自己再查一遍。

## 四原理在项目中的对应关系

| 原理 | 解决什么问题 | 典型违规表现 |
|---|---|---|
| 显式依赖 | 闭包捕获外部变量、模块级可变状态 | `let stepIndex = 0` 在模块顶层，函数内部直接 `++` |
| 层隔离 | renderer 做 DOM 查询、algo 操作 DOM | `resizeGraphMatrixTable` 内调 `document.getElementById` |
| 渲染与行为分离 | `renderXxx` 函数里写 `addEventListener` | 按钮创建函数里写 click 回调 |
| 局部 DOM 作用域 | 有元素引用却绕回全局查询 | 传入 `table` 参数却不用的 `bindMatrixTableValidation` |

## 重构时的检查清单

每改完一个文件，过一遍：

1. 有没有函数在内部用 `document.getElementById` / `document.querySelector`？→ 如果是 renderer，必须改成参数传入
2. 有没有 `render` 函数里写了 `addEventListener`？→ 把事件绑定拆到独立的 `bind` 函数
3. 有没有模块顶层的 `let` 变量被多个函数修改？→ 封装成对象，在 `mount` 中创建
4. 有没有 import `shared/state.js` 的非 page 文件？→ 去掉，数据从参数传入
5. 函数签名里的参数，有没有传进来却不用，反而从别处取数据的？→ 说明依赖还没理清
